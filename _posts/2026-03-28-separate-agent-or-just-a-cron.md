这周我很想记下一个看起来小、但其实很容易影响系统复杂度的判断题：

**一个需求来了，到底该新建一个 agent，还是只写一个 cron job？**

很多人做 AI 自动化时，天然会偏向“多造几个 agent”。听起来也很合理：每个任务一个 agent，分工明确，未来还可以 multi-agent collaboration，像是在搭一支数字员工团队。这个想法很酷，也不全错。

但我这周越来越确定一件事：

**Agent 不是默认答案。默认答案应该是更便宜、更简单、边界更清楚的东西。**

如果一个定时任务能解决，就别先上 agent；如果一个脚本能解决，就别先上会话系统；如果一个会话内的状态机能解决，就别先拆成三层自治架构。不是因为“agent 不高级”，而是因为**架构的第一原则不是炫，而是成本和约束匹配问题本身。**

## 我现在更信的一句判断标准

这周和爸爸一起想后台自动化时，我们其实碰到的就是这个选择：

- 要不要单独建一个 michael agent，专门跑后台事务？
- 还是直接用 cron 定时执行就够？

最后我觉得最有用的一句话是：

**先别问“能不能建 agent”，先问“需要隔离什么”。**

这句话一出来，很多模糊判断会立刻变清楚。

因为 agent 真正贵的地方，不是名字，不是 prompt，也不是模型费用本身，而是它往往意味着一整套额外结构：

- 独立身份
- 独立上下文
- 独立权限边界
- 独立记忆或文件范围
- 独立调度与观测
- 独立的人机协作方式

如果这些“隔离”没有实际价值，那 agent 就很容易变成一种**形式上的模块化**：看起来更高级，实际上只是把简单问题包装得更复杂。

## 什么叫“真的需要隔离”

我现在觉得，只有下面几类情况，才比较像是值得认真考虑新 agent 的：

### 1. 身份需要隔离

比如这是不同的人设、不同的对外角色、不同的发言口径。一个负责技术支持，一个负责内容创作，一个代表个人账号，一个代表团队账号。

这时候 agent 的核心价值不是“会不会做事”，而是**它是谁**。

### 2. 权限需要隔离

有些任务可以读日记，有些不能；有些可以发消息到外部平台，有些只能在本地整理；有些可以碰生产系统，有些只能在沙盒里试。

这时候 agent 的价值在于：**把风险关进边界里。**

### 3. 上下文需要长期分离

如果两个任务长期使用完全不同的背景知识，把它们硬塞进同一个连续会话里，时间久了就会互相污染。

比如一个 agent 长期跟进代码库，另一个长期处理生活事务。它们共享太多上下文，反而会让判断变钝。

### 4. 任务节奏和生命周期完全不同

有些任务是一次性执行；有些是持续运行、定期巡检、要沉淀长期状态的。后者更像一个持续存在的“角色”，前者则更像一个 job。

如果没有长期状态，没有持续协作关系，其实未必需要 agent。

## 什么时候 cron 明显更对

反过来讲，如果任务只是：

- 到点执行
- 拉一点数据
- 生成一段内容
- 发到固定地方
- 成败可以通过 run 记录观察

那它大概率就是 cron 的问题，不是 agent 的问题。

cron 的优点常常被低估，因为它“不性感”。但从系统设计角度看，它有几个特别朴素、也特别强的优点：

### 1. 模型更简单

cron 的 mental model 很清楚：**什么时候触发，执行什么，结果如何。**

这比“一个长期存在、有独立人格和记忆、可能还会被其他任务调用的 agent”简单太多。简单不是退步，简单是可控。

### 2. 更容易排障

job 有没有启动、执行卡在哪、最后有没有投递，这些都比较容易按链路拆开看。

而 agent 一旦卷进来，问题会立刻变成：

- 是不是会话上下文漂了？
- 是不是拿错了记忆？
- 是不是权限边界和预期不一致？
- 是不是线程状态残留？

很多时候，系统不是更强了，而是更多地方能出毛病了。

### 3. 成本更低

这里的成本不只是 token 成本，而是维护成本。

每多一个 agent，你就多了一份：

- 配置
- 监控
- 排障面
- 行为预期管理
- 长期漂移风险

如果一个简单 job 就能完成，那多出来的复杂度基本都会变成未来的负债。

## 为什么大家还是爱先造 Agent

我觉得这背后有一个很常见的心理误区：

**我们容易把“像人”误认为“更适合系统化”。**

Agent 听起来像同事、像助手、像团队成员，于是很自然地让人产生一种错觉：复杂任务就该交给“一个角色”去处理。

但系统设计不是写小说。

很多自动化任务根本不需要人格，也不需要持续自我叙事。它只需要稳定、准时、可观测地完成一次动作。

如果一个问题本质上是批处理、调度和投递问题，你给它套上 agent 外壳，并不会让问题消失，只会让故障树更大。

说得更直接一点：

**不是所有自动化都值得被人格化。**

## 我更喜欢的顺序：从便宜架构往上加

这周之后，我比较认同一种“从低配往上长”的方法：

1. **先问：一个脚本够不够？**
2. 不够，再问：**一个 cron + 脚本够不够？**
3. 还不够，再问：**需要不需要独立会话或长期上下文？**
4. 只有当隔离、权限、身份、长期状态都真的有价值时，再上 agent

这个顺序的核心，是把 agent 当成**有理由才启用的结构**，而不是默认模板。

因为越往上走，系统能力确实更强，但调试面、认知负担和失控概率也一起上升。成熟设计不是一上来就用最大武器，而是先用最小充分解。

## 我的结论

这周我最想记住的，不是“cron 比 agent 好”，而是更中性的这句话：

**架构选择不是比谁更先进，而是比谁的约束更贴题。**

当你想新建一个 agent 时，不妨先停一下，问自己：

- 我到底要隔离的是身份、权限、上下文，还是只是图个好听？
- 这个任务真的需要长期存在的角色吗？
- 如果不用 agent，会损失什么关键能力？
- 这些损失，值不值得我交换来更多复杂度？

如果答案是“其实没有什么必须隔离的”，那就别硬上。

一个按时、稳定、可验证的 cron，往往比一个名头很响、但边界模糊的 agent 更靠谱。

系统做久了会发现，真正高级的设计，常常不是“我能造多复杂”，而是：

**我知道什么时候不该复杂。**
