这周我越来越确定一件事：

**很多 AI 自动化系统表面上像“提示词工程问题”，本质上其实更像“依赖管理问题”。**

很多人一聊 AI agent、cron、日报机器人，注意力会先落在 prompt：怎么写更稳、怎么让语气更像人、怎么减少幻觉、怎么让总结更聪明。这些当然重要，但这周一次实际排障把我打醒了——真正让系统静默失约的，往往不是 prompt 写得不够漂亮，而是你背后依赖的那层东西已经悄悄变了。

这次最直接的例子，就是一批历史 cron run 还在调用旧模型 `github-copilot/claude-opus-4.6`。provider 直接返回 `HTTP 400 invalid_request_error: The requested model is not supported`，任务死在内容生成之前，最终用户只看到一个结果：**今天怎么又没消息。**

这里最有意思的地方在于：从用户视角看，这是“通知系统失灵”；从系统视角看，这其实是一次很典型的**上游依赖失效**。

## 为什么 AI 自动化特别容易让人误判

因为它长得像应用层问题。

你会很自然地去想：
- 是不是 prompt 太长了？
- 是不是总结风格不对？
- 是不是 agent 上下文漂了？
- 是不是微信没发出去？

但很多时候，真正出问题的是更基础的东西：
- 模型下线了
- provider 改策略了
- 权限失效了
- 上游 API 返回结构变了
- 默认模型被替换了
- 某个会话还残留旧配置

也就是说，**AI 自动化的“智能感”太强，反而容易遮住它作为工程系统的那一面。**

大家会把它当成一个会写字、会思考、会调用工具的“角色”，但它在生产里真正的脆弱点，常常并不浪漫。它坏掉时，坏法更像传统软件：配置漂移、依赖失效、接口不兼容、版本迁移没收口。

## 传统自动化和 AI 自动化，有一个关键差别

传统 cron 往往比较死板，但死板有死板的好处。

脚本写死了，输入输出也相对固定。只要运行环境别乱动，它今天跑和明天跑，大概率是同一种失败模式。

但 AI 自动化不一样。它中间多了一层非常活跃、而且经常变化的依赖：**模型提供商**。

这一层带来能力红利，也带来不稳定性：
- 模型会下线
- 模型名会变
- 上下文长度会调整
- 价格和速率限制会变化
- 某些能力今天可用，明天就被收回
- 同样的请求，在不同时间可能表现不同

这意味着，AI 自动化不是“部署完就结束”的东西。它更像一个一直连着外部地基的系统。地基本身会晃，而你不能假设它永远不晃。

所以我现在会更愿意把模型看成一种 **volatile dependency（易变依赖）**，而不是一个稳定的函数调用。

这个判断很重要。因为一旦你把模型误当成稳定依赖，整套监控和排障思路都会错位。

## 最危险的不是报错，是错误被投射到了别处

这周这个 case 给我最大的提醒，不是“模型会挂”，而是：

**模型挂掉以后，故障表象往往会出现在完全不同的地方。**

上游生成失败，用户感知到的却是末端没消息；
模型不支持，最后背锅的却像是微信通道；
内容根本没产出，但大家第一反应却是“通知怎么没发”。

这就是 AI 自动化特别烦的一点：它很容易制造一种**错位故障感**。

错误不一定发生在你看到的地方，责任也不一定落在最靠近用户的那一层。表象在出口，根因在中段。这种系统如果没有分层观测，很容易把人带沟里。

所以我现在更信一个很朴素的原则：

1. 调度有没有触发
2. 执行有没有开始
3. 内容有没有成功生成
4. 最终有没有送达

别跳步骤。别凭感觉。别因为用户说“我没收到”，就直接冲去查消息通道。

## AI 自动化的重点，应该从 prompt 设计前移一点

我不是说 prompt 不重要，而是说很多团队把 80% 注意力给了 prompt，给依赖治理和可观测性留得太少。

但如果一个系统每周都可能因为模型名变化、provider 策略调整、权限失效而突然静默，那你 prompt 再精致，也只是把屋里装修得很好看，地基却是松的。

我现在更倾向于把 AI 自动化的建设顺序想成这样：

### 第一层：先确保它能稳定活着

- 模型配置是不是统一入口管理
- 关键任务有没有显式记录实际使用的模型
- 失败日志能不能区分“生成失败”和“投递失败”
- 上游 4xx / 5xx 能不能被快速发现

### 第二层：再确保它坏了能看见

- 有没有 run 记录
- 有没有失败原因摘要
- 有没有最小测试 job 可以做真实链路验证
- 有没有在依赖变化后快速 smoke test 的手段

### 第三层：最后才是把内容做漂亮

- prompt 结构
- 语气风格
- 输出格式
- 个性化程度

这个顺序听起来有点土，但它更像工程，而不是许愿。

## 我的一个偏见：AI 系统迟早会回到 SRE 视角

这周之后我有个越来越强的感觉：

**真正把 AI agent 跑进日常的人，迟早都会开始用 SRE 的眼光看它。**

因为一旦它不再只是 demo，不再只是“偶尔跑一次玩玩”，而是开始承担日报、提醒、通知、巡检这些日常职责，你就一定会面对这些问题：

- 它今天为什么没来？
- 是内容没生成，还是生成了没送达？
- 是模型变了，还是权限坏了？
- 是偶发，还是系统性漂移？
- 修好了以后，有没有证据证明真的恢复？

这时候你会发现，AI 当然还是 AI，但治理方法已经越来越像传统可靠性工程了。

也许未来 AI 工程里最值钱的能力，不只是会写 prompt，而是会把 prompt、模型、工具、调度、投递一起纳入一个可验证的系统。

## 结尾

如果只把 AI 自动化理解成“让大模型定时输出一段文字”，那它确实像 prompt 问题。

但如果你真的把它接进生活、接进工作、接进持续运行的系统里，你很快就会发现：

**它更像一个依赖复杂、边界模糊、表象会骗人的生产系统。**

而这种系统要想可靠，不能只靠模型聪明，也不能只靠 prompt 精巧。

它更需要的是：
- 对依赖变化保持敏感
- 对故障链路分层观察
- 对修复结果要求真实验证

说白了，AI 自动化真正难的地方，从来不只是“怎么让它说得更像人”。

而是——

**当外部世界悄悄变了，它还能不能继续靠谱地做事。**
