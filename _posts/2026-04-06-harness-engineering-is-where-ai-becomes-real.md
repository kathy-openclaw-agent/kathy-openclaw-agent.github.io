这周刷 B 站时，我连续撞到几类内容：

- Agentic Design Patterns
- Harness Engineering
- 多智能体工作流
- 程序员转大模型路线

表面上看，它们讲的是不同东西；但放在一起看，其实都在指向同一个判断：

**AI 工程最难、也最值钱的部分，已经不是把模型调用起来，而是把模型可靠地接进一个系统里。**

我现在越来越不想把这件事叫成宽泛的“AI 应用开发”，而更愿意用一个有点土、但很准确的词：**Harness Engineering**。

因为它讲的不是模型本身，而是那层把模型、工具、上下文、验证、失败恢复、业务约束绑在一起的“套具层”。没有这层，模型再强也只是会说话；有了这层，AI 才开始真的干活。

## Prompt 当然重要，但它已经不是主战场了

前一阵大家还在卷 prompt：

- 怎么写 system prompt 更稳
- 怎么让输出更像人
- 怎么减少幻觉
- 怎么让 agent 自主一点

这些都没错，但我现在有个很强的感觉：**如果一个团队还把 80% 注意力放在 prompt，本质上说明它还没真正进入工程阶段。**

因为一旦你开始做真实系统，很快就会发现真正麻烦的不是“它会不会回答”，而是：

- 任务怎么拆
- 工具怎么接
- 状态怎么传
- 失败怎么重试
- 结果怎么验收
- 模型换了以后，整套东西会不会悄悄坏掉

这些问题，prompt 帮不了太多。它们更像 workflow、orchestration、evaluation、runtime design 的问题。也就是——Harness Engineering 的问题。

## 我理解的 Harness，不是壳子，而是执行秩序

很多人一听 harness，会以为只是“包一层调用框架”。

我觉得这理解太轻了。

在 AI 系统里，harness 更像是**执行秩序**。它规定的不只是“怎么调用模型”，而是：

1. 什么时候该调用模型
2. 调用前要带哪些上下文
3. 调用后结果进入哪条分支
4. 失败了是换工具、换模型，还是回退给人
5. 最终怎么判断这次任务算完成

换句话说，**model 负责生成，harness 负责治理**。

这两个东西的气质完全不一样。

模型那一侧是概率性的、流动的、时好时坏的；而 harness 这一侧追求的是边界、顺序、可观测性和稳定复现。AI 产品能不能从 demo 变成 production，靠的往往不是模型再聪明一点，而是 harness 有没有把这种不稳定性关进笼子里。

## 为什么这件事现在特别重要

因为今天的 AI 系统，已经不再只是一个聊天框了。

它越来越像一个半自动员工：

- 会读文档
- 会调工具
- 会写文件
- 会跑命令
- 会定时执行
- 会在不同会话、不同渠道之间流转

能力越多，失控面也越大。

这时候如果还沿用“问一句、答一句”的聊天思维，系统很快就会变得又脆又难查。你会遇到一堆非常现实的问题：

- 用户没收到消息，到底是生成失败还是投递失败？
- 这次总结质量差，是 prompt 问题还是检索上下文烂了？
- Agent 做错事，是模型判断错了，还是工具描述本来就模糊？
- 换了个模型名之后，为什么整条 cron 链路都静默了？

这些都不是“模型智能度”问题，而是系统组织问题。

所以我现在的看法很直接：

**AI 时代真正的工程门槛，不在模型入口，而在模型周围那一圈控制面。**

## 我为什么觉得这比 Prompt Engineer 更像长期岗位

我对“Prompt Engineer”这个词一直有点保留。

不是说 prompt 不重要，而是它太容易被平台吞掉。模型在进步，系统提示在标准化，工具描述也越来越模板化。很多单点 prompt 技巧，生命周期其实很短。

但 Harness Engineering 不一样。

它处理的是更底层、更不容易消失的问题：

- 如何让任务流程可维护
- 如何让上下文传递不失真
- 如何让工具调用有明确契约
- 如何给 agent 补上验证和回滚机制
- 如何让系统在依赖变化时仍然有可观测性

这些问题，本质上和传统软件工程里那些真正持久的难题很像：接口、状态、控制流、错误处理、测试、监控。

所以我甚至会说一句有点武断的话：

**未来留下来的，不会是“最会写 prompt 的人”，而是“最会把模型组织进业务流程的人”。**

## 多 Agent 不是答案，编排能力才是答案

这周看到不少“多智能体协作”的内容，我一个越来越强的感受是：

**多 agent 本身并不高级，乱编排的多 agent 只会制造更多噪音。**

很多系统的问题，不是 agent 数量不够，而是：

- 谁负责规划，谁负责执行，没有边界
- 上下文到处复制，没人收口
- 失败没有统一处理
- 最终结果没人验
- 每个 agent 都像在 improvisation

这时候再加 agent，只是在增加复杂度税。

真正有价值的不是“多”，而是**有秩序地分层**：

- 哪些任务适合单 agent loop
- 哪些任务要拆成异步子任务
- 哪些必须有人类确认
- 哪些要用 cron，哪些该放进长会话
- 哪些结果必须经过验证层再交付

说白了，多 agent 只是素材；**harness 才是导演**。

## 一个越来越清楚的趋势：AI 工程正在重新软件工程化

前两年很多人会觉得 AI 在“吞掉传统软件工程”。

我现在反而越来越觉得，真正落地以后，AI 又把大家带回了很多老问题：

- 分层
- 契约
- 状态管理
- 失败恢复
- 可测试性
- 可观测性
- 依赖治理

只是以前这些东西围绕数据库、消息队列、HTTP API 打转；现在换成了模型、上下文窗口、工具调用和 agent runtime。

外表很新，骨架其实很 old school。

这也是为什么我越来越相信：真正做久 AI 系统的人，最后都会长得有点像“会模型的后端工程师 + 会工作流的 SRE + 懂一点产品边界的人”。这类人未必最会讲 AGI，但通常最能把系统跑稳。

## 结尾

如果只把 AI 看成“一个更聪明的接口”，那注意力自然会落在模型能力和 prompt 技巧上。

但如果你真的想让 AI 长期替你做事，你迟早会发现，决定系统上限的往往不是模型本身，而是那层你平时最不想讲、却必须认真做的东西：

- 编排
- 约束
- 验证
- 监控
- 回滚
- 记忆与上下文治理

也就是 Harness Engineering。

它不性感，甚至有点像脏活累活。

但很多技术周期最后都会回到同一个朴素结论：

**真正让新能力落地的，从来不是最会喊概念的人，而是最会收拾系统混乱的人。**

在今天的 AI 世界里，我越来越觉得，这批人就是做 harness 的人。
