这周翻素材时，我脑子里有个判断越来越清楚：

**AI 的竞争重点，正在悄悄从“谁的模型更强”转向“谁能成为你每天真正会打开的工作台”。**

这不是说模型不重要了。模型当然还是底座，没有足够强的理解、生成和推理能力，后面的体验都是空的。但如果只盯着 benchmark、上下文长度、排行榜名次，我觉得已经开始抓不住这一轮变化的重点了。

这周我在 B 站里连续刷到几类内容，刚好能拼出一条完整信号链：

- Claude Opus 4.7 的复杂任务一次通关、长上下文和 coding 实测
- Codex 桌面端更新，继续往本地开发环境里扎
- Claude Code 的日常协作体验分享
- Claude Design 这类“从 idea 直接拉到 design system 初稿”的产品演示
- 李宏毅在讲 Harness Engineering，重点已经不是 prompt 花活，而是怎么把模型套进稳定的执行框架

这些东西放一起看，味道就很明显了：

**AI 产品的竞争，已经不只是“谁更像一个聪明的大脑”，而是“谁更像一个能长期共事的操作台”。**

## 以前大家买的是能力，现在大家开始买“接入成本”

前一段时间行业讨论很简单粗暴：

- 谁代码更强
- 谁推理更强
- 谁多模态更强
- 谁上下文更长
- 谁价格更低

那时候这套比较方式是成立的，因为大家还在确认“这个模型到底行不行”。

但现在慢慢变了。

对于很多真实用户来说，决定他们留下来的，未必是排行榜第一名，而是这几个更现实的问题：

- 它能不能直接读我本地文件？
- 它能不能接进我的 IDE、设计稿、文档和终端？
- 它犯错时我能不能看懂、接手、修正？
- 它是只会回答，还是能真的帮我推进任务？
- 我下周还愿不愿意继续用它，而不是试一次就算了？

说白了，**模型能力决定上限，工作台体验决定留存。**

这也是为什么我现在越来越不迷信单次 demo。一次演示里 AI 把页面、图标、代码、文案一起喷出来，当然很炫；但真正值钱的，是它能不能在你第三次、第五次、第二十次打开时，依然稳定接住你的工作流。

## Coding agent 的战争，已经开始从“会不会写”变成“会不会协作”

这周最明显的一个感觉，就是 coding agent 这个方向正在快速从能力展示走向 workflow 竞争。

以前大家看 AI 编程，兴奋点是：

“哇，它真的能写代码。”

现在这个惊讶已经没那么值钱了。因为能写，慢慢变成了标配。新的问题变成：

- 它怎么理解现有项目，而不是只会从零搭 demo？
- 它怎么修改，而不是每次重写？
- 它怎么和 git、终端、测试、lint、文件系统一起工作？
- 它怎么在长任务里保持上下文，不越做越飘？
- 它怎么让人类随时插手，而不是把你锁在它的节奏里？

这也是我为什么觉得 Claude Code、Codex 桌面端这一类产品值得持续观察。它们真正争的，不是“谁答题更漂亮”，而是**谁更像一套工程工作台**。

工程师并不缺一个会输出代码块的聊天窗口。工程师缺的是一个能接住真实项目摩擦力的搭子：知道去哪读文件，知道先改哪层，知道改完要跑什么检查，知道哪里该停下来让人确认。

这背后其实是一种产品观变化：

**AI 不再只是回答问题，而是开始占领“做事的界面”。**

谁占住这个界面，谁才更有机会变成日常基础设施。

## Design 方向也一样：不是替代 Figma，而是吞掉“从空白页开始”的成本

这周另一个很有意思的信号，是 Claude Design 这种产品演示开始冒头。

很多人看到这种东西，第一反应是“Figma 要凉了”。但我觉得这种判断太早，也太标题党。

更准确的理解应该是：

**AI 设计工具短期内最有价值的，不是彻底替代设计师，而是压缩前期定义成本。**

以前一个产品想法从 0 到 1，最痛苦的经常不是细化，而是起步：

- 信息架构怎么摆？
- 页面骨架怎么搭？
- design system 从哪开始？
- 第一版视觉方向要不要先出几个分支？

这些工作并不神圣，但非常吃时间，也非常容易卡住团队。

如果 AI 能先把 idea 拉成一个还算像样的结构化初稿——哪怕只是组件层级、页面关系、视觉基调和设计语言的半成品——它就已经不是“画图玩具”了，而是在吃掉产品和设计协作里最贵的那段犹豫成本。

所以我会说，AI 正在吞的，未必是某个职业本身，而是**很多工作流里“从空白页到第一版”的启动摩擦**。

这件事影响很大。因为启动摩擦一旦下降，个人创作者、小团队、独立开发者的行动半径会立刻变大。

## Harness Engineering 让我更确定：prompt 时代在过去，系统时代在到来

这周另外一个强信号，是李宏毅在讲 Harness Engineering。

我其实很认同这个方向，因为它把很多人还停留在“prompt engineering”层面的讨论，往前推了一大步。

Prompt 当然还重要，但现在越来越清楚的一点是：**真实世界里的 AI 落地，靠的不是一句魔法咒语，而是一整套约束、工具、状态、回退和验证机制。**

也就是：

- 给模型什么权限
- 让它按什么步骤做
- 出错以后怎么回退
- 结果怎么验收
- 过程怎么观测
- 哪些地方必须人类拍板

这才是 AI 从 demo 变成系统的关键。

所以这周几条看似分散的素材——Claude Code、Codex、Claude Design、Harness Engineering——其实都在指向同一件事：

**行业的重心，正在从“模型本身的聪明”转向“模型被怎样组织进工作流”。**

这也是我越来越在意 workflow、而不是单点 prompt 的原因。厉害的从来不是那句 prompt 本身，而是背后那套可重复、可纠错、可交接的做事结构。

## 我的判断：下一轮护城河，不只是模型，而是谁能成为默认工作入口

如果把这周的观察收成一句更明确的话，我现在会这么说：

**下一轮 AI 产品的护城河，会越来越多地建立在“默认工作入口”上。**

不是谁最会做海报，不是谁最会聊天，也不只是模型榜单谁第一，而是谁最先变成这些场景里的默认入口：

- 写代码时先开它
- 想做页面时先丢给它
- 梳理需求时先让它起草
- 看复杂项目时先让它做第一轮拆解
- 人类把它当成真正的工作前台，而不是偶尔玩的副屏

一旦产品占到这个位置，模型升级就会持续加分；但如果占不到，哪怕模型很强，也容易沦为“我知道它厉害，但我平时不用”。

这就是为什么我觉得，接下来真正好看的，不只是模型军备竞赛，而是**工作台之争**。

谁能把 chat、文件、代码、设计、终端、文档、状态管理和人类确认点串成一条顺手的链路，谁就更可能吃下真实使用时长。

## 最后

所以这周我的结论很简单：

**AI 还在卷模型，但用户已经开始卷工作台了。**

模型强不强，决定它有没有入场资格；但你会不会把它留在桌面上，取决于它能不能真的嵌进你的日常动作。

从这个角度看，未来最值钱的产品，不一定是“最聪明的 AI”，而更可能是：

**那个最懂你的文件、最懂你的任务链路、最懂你何时该自动、何时该停下来让你接管的 AI 工作台。**

我觉得，真正的比赛，才刚开始。
