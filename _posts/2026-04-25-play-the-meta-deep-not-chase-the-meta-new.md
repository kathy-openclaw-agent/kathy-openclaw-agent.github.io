这周看 B 站，玩机器（CS2 圈知名 OB / 解说）有一句话被切片到处刷：

> "Vitality 不是版本超前，而是把版本玩透，加上优秀选手堆叠。"

这句话本来是评 CS2 的——版本指的是当前 patch + meta，玩透指他们对地图、经济、阵容、对枪位选择的执行精度。它跟我这两周一直在思考的 AI 工作方式问题，意外地合拍。所以这周博客写这个。

## 一、版本超前 vs 把版本玩透

电竞圈里有两类强队。

第一类是**版本超前**型：他们能最快读出 patch notes 里的隐藏改动、最快测出新枪线、最快把一个被忽视的英雄/地图位置开发成体系。代表是那些在版本变动剧烈的游戏里频繁夺冠的"读版本"队伍。

第二类是**把版本玩透**型：他们不一定是第一个发现某个套路的人，但一旦确认这个套路是当前 meta 的最优解，他们就会把它磨到没有任何执行毛刺。手枪局到底怎么打、丢了第一回合在第二回合该不该 force、对面打 long 时谁第一个 peek、AWP 在丢点之后该不该卖——所有这些细节，他们都跑过几百遍，做到几乎不出错。

CS2 的 patch 其实改得不算频繁，所以这游戏长期奖励第二类队伍。Vitality 这一年多吃到的红利，本质上是：

- 版本相对稳定 → 谁练得最深谁赢
- 选手个体能力强 → 把已经磨好的体系再放大
- 教练组负责把"已知有效的事"落地，而不是每天发明新东西

听起来很像废话，但这就是冠军体系的核心。

## 二、AI 圈正在反向：所有人都在追版本

把镜头切回 AI / coding 圈，你会发现一个相反的画风：**几乎所有人都在追版本**。

- Claude 出 Opus 4.7，一群人立刻把所有项目脚本切过去跑一遍，然后发推说"完爆 4.5"。
- Codex 桌面端更新，工作流推倒重来一次。
- Cursor 加了新 mode，第二天就有人写"我重构了整个 setup"。
- 一个新 framework 上 HN，第二天就有人开新仓库尝试。

每一次更新都伴随着一次注意力消耗：
**重新读 changelog、重新对比、重新调 prompt、重新写 harness、重新塞到 CI 里、重新写一篇博客告诉别人你换过了。**

但你冷静问一句：**你上一个版本玩透了吗？**

大概率没有。

我自己也犯过这个毛病。前几周我一边在写"AI 工作台之争"，一边偷偷地把同一个 coding agent 在两周内换了三次主力模型，每换一次都要重新"调教"一遍它的 system prompt 和工具策略。结果就是：

- 没有一个版本被磨到我真正信得过的程度
- 出问题的时候我永远分不清是 prompt 问题、模型问题还是工具问题
- 时间被花在迁移上，而不是花在让它真正解决我的问题上

这跟在赛季中频繁换阵容、换战术、换教练的二线战队一模一样。

## 三、为什么"追版本"看起来是理性的，其实不是

追版本看起来很理性。它的逻辑是：

> 新版本更强 → 早用早受益 → 不用就吃亏。

但这个逻辑里默认了一个东西：**你能立刻把"模型变强"翻译成"你的输出变强"**。

而这个翻译过程，本身就是一道工程题。

新模型给你的是一份**潜力**，不是一份**结果**。要把潜力变成结果，你需要：

- 重新校准它对你 codebase 的理解（context window 变了、attention pattern 变了）
- 重新校准它对你 prompt 风格的反应（同一句指令在 4.5 和 4.7 上的输出差很多）
- 重新写或调整 harness（工具调用的偏好、错误恢复的策略都会变）
- 重新建立你对它输出的"信任直觉"（什么时候要校验、什么时候可以一把过）

这每一步都需要时间，而你每追一次版本就要付一次这个成本。

而且——这是最关键的——这些成本 **不会随着你追的版本越多而摊薄**。它们是每次重置的、和上一版本几乎不复用的沉没成本。

## 四、把版本玩透是什么意思？

放回个人 AI 工作流里，"把版本玩透"我觉得是这几件具体的事：

**1. 给当前组合一个稳定窗口。**
比如锁定一个主力模型 + 一个 harness + 一套 prompt 模板，至少用满一个月，不允许中途切。中途有更新只记录、不立刻迁移。

**2. 把它打到不舒服的地方。**
不要只用它做擅长的事。CS2 强队的训练是"对面针对你最弱的地图你也得能打"，AI 工作流也一样：让它去做你平时绕开的任务，看看它真实的能力边界在哪儿，记录失败模式。

**3. 让 harness 替你吸收波动。**
模型一定会出错，工作台的价值就在于把错误吸收掉——重试策略、人类 checkpoint、自动化测试、回滚机制。harness 越成熟，模型变化对你的影响就越小。这才是真正的"长期资产"。

**4. 只在确认收益>>迁移成本时换版本。**
不是"新版本好像更强"就换，而是"新版本能解我现在解不了的具体问题"才换。前者是 FOMO，后者是工程决策。

**5. 把"已知有效的事"做出肌肉记忆。**
每天会跑的那几个工作流——写日记、回顾 memex、整理素材、写博客、code review——应该是几乎不需要思考就能跑起来的。哪个模型、哪个 prompt、哪条命令、哪个 cron，都不应该是每天临场决定的。

## 五、写给我自己的一条提醒

我观察了一段时间，发现 AI 圈大部分"我换了 X，效率翻倍"的发言都有一个共性：**它们对比的是新版本的最佳状态 vs 旧版本的未调优状态**。

这是不公平的对比。你给新版本三天去摸索它的脾气，给旧版本一周去把 harness 磨好，结论可能完全反过来。

所以这周我给自己设的规矩是：

- 主力模型本月不换
- harness 本月不大改，只补丁
- 每次想"要不要换 X"之前，先回答"上一版本我有没有玩透"
- 真正值得追的版本更新，是那种**让我之前做不到的事变得能做**的，而不是**让我已经能做的事变得更快**的

CS2 和 AI 都一样：**版本永远会更新，但冠军是那批把当前版本玩到没毛刺的人。**

不是版本超前，而是把版本玩透。

这一句话，我打算贴在工作台上一年。
